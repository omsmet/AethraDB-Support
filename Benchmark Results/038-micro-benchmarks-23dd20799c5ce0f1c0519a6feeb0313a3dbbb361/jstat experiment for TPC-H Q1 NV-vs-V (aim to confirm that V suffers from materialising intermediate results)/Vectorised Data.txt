Conclusions (Based on visualisation tool):
- Eden capacity remains is upgraded once and its usage gradually increases
- Survivor space 0 is not used (S0C = S0U = 0), but survivor space 1 is used (S1C is upgraded to 16384 and S1U varies around 8300)
- Old space is also used (OC downgraded once and OU steps from 0 to 1467 once)
- Meta space also used (MC increases to 2688 and MU to 2477.2)
- Compressed class space is used (CCSC and CCSU respectively are increased to 320 and 220)

- Young (Light) garbage collection is run 6 times:
    - YGC (# young garbage collection events) is 6 at the end
    - YGCT (young garbage collection time) 23ms
- Full garbage collection never runs:
    - Makes sense as old space is never filled up (OU always < OC)
    - FGC (# full garbage collection events) remains 0
    - FGCT (full garbage collection time) remains 0
- GCT (total garbage collection time) is 23ms

The fact that young garbage collection is run is a bit surprising: the size of the heap is definitely large enough to account for e.g. allocated arrays and such.
Upon inspection we find:
- As expected, for the non-vectorised execution, the garbage collector never kicks in, as all allocated objects fit within the original Eden space.
- However, for the vectorised execution, young garbage collection is executed 6 times (full garbage collection never kicks in).
    I found this quite surprising as the only difference between both implementations should be the allocation of a couple of intermediate result vectors, accounting for something like 10MB of memory.
- Upon closer inspection, I found that jstat additonally reveals very high memory usage:
    upto ~10GB before garbage collection is performed, while it shows around 500MB for the non-vectorised case for the whole query
This thus pointed to something very wrong with the implementation: it seems somehow I somehow did not fully implement the arrow optimisation where we copy a fixed-length binary column into a buffer byte[],
instead of relying on the arrow driver to allocate a new array for every element in such a column , in the vectorised paradigm
